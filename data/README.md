# `data/` â€” the dev registry seed

These YAML files are the **Git-managed control plane that seeds the governed
Registry** during development. They are the current file form of what
[`CONTEXT.md`](../CONTEXT.md) calls the *Git-versioned registry (the governance
truth)*: objects enter only through this Git seam, **never a mutable store**.

What "governed" means here: every manifest is parsed and pushed through the
`@atlas/schema` gate (`validateRegistryManifest`) before it counts. Invalid data
fails fast with `file + id + reason`. The schema (types/constraints) lives in
`@atlas/schema`; only the **instances/content** live here.

## What this is NOT

- **Not the production source of truth.** Only the **dev adapter**
  (`context-layer/src/adapters/dev/`, `portal/src/adapters/dev/`) reads these
  files. Core depends only on the read-only `Registry` / `SourceContentProvider`
  / `AvailabilityProvider` ports (`context-layer/src/registry/registry.ts`) and
  is unaware of where records come from.
- **Not wired into prod at runtime.** The composition root
  (`context-layer/src/composition.ts`) is the single seam that swaps the dev
  factories for live adapters. In production the governed Registry is assembled
  by the live discovery adapter
  ([`plans/017`](../plans/017-live-confluence-discovery.md)); these YAML files do
  not participate. Per `seed-dev-adapter-principle`: the design target is the
  real-environment live fetch; `data/*.yaml` is dev mock only, never the design
  standard.
- **Not hand-tuned long-term.** After the decontamination migration
  ([`plans/016`](../plans/016-domain-decontamination.md)) the seed is meant to be
  **regenerated by the model**, not hand-edited. Coverage/line-count is not a
  design axis.

Missing manifests are an **honest gap, not a crash**: loaders contribute an
empty registry rather than throwing (malformed data still fails fast).

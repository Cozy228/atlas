## 0. 怎么用这份文档

**血缘说明（重要）**：`18844da` 不是 `feat/0.2.0` 的线性祖先——两者在 `bec3f16`(2026-05-07) 分叉，各自 175 / 242 commit。但 `18844da` 持有**旧架构**（`contextBundleService`/`Topic`/`Anchor`/`pilotRegistry` 都还在，在 0.2.0 被删），所以它是"旧接口的参照快照"。真实旧实现从 **stash / local-dev 分支** 取；每个端口的旧签名，对照 `18844da` 再核一遍即可。

**两层迁移策略**：

- **Tier A — 直接采用 0.2.0（generic，无特定业务逻辑）**：schema、resolver 端口类型、resourceContextService 投影逻辑、cache、discovery/derive 管线骨架、composition 骨架、index 布线。这些新实现已完整，**照搬 + 改 import**，不需要"翻译旧实现"。见 §2。
- **Tier B — 把旧实现的真实逻辑翻译进新端口（每端口一个 prompt）**：真实解析逻辑（Confluence 各页面、Terraform README、release notes）、规范值（kernel 分类规则、LZ 拓扑）、真实抓取/鉴权（Confluence/TFE 端点与认证）。见 §3。

**每个 Tier B 端口的循环**：
1. 冻结新契约（端口接口 + schema 类型，本文件已给 verbatim）。
2. 从 stash / local-dev 分支取出该端口的旧实现文件，连同 §3 对应 prompt + 0.2.0 新文件一起喂给模型。
3. **行为级校验**：跑新契约的 fixture/golden test（`devMocks/` + `*.golden.test.ts` 已定义期望形状），别只看"能编译"——拍平的 schema 会静默丢语义。

**依赖顺序**（先做地基，后做上层）：
```
Tier A: schema → resolverTypes → cache
Tier B: P1 Confluence 抓取substrate → P2 guidance parse
                                    → P3 availability parse → P8 availability resolver
                                    → P4 reference discovery
        P5 Terraform provider（独立）
        P6 kernel 规范值（独立）
        P7 landing zones（独立，喂给 discovery）
        P9 release notes（独立）
最后: discovery/derive 管线 + composition 布线（Tier A 骨架 + P6/P7 值）
```

## 1. 全局变更（一次性、到处适用）

### 1.1 环境变量改名（去掉 `ATLAS_` 前缀，按来源系统命名）

| 旧 | 新 |
|---|---|
| `ATLAS_CONFLUENCE_TOKEN` / `_BASE_URL` / `_EMAIL` | `CONFLUENCE_TOKEN` / `CONFLUENCE_BASE_URL` / `CONFLUENCE_EMAIL` |
| `ATLAS_TERRAFORM_TOKEN` / `_BASE_URL` | `TERRAFORM_TOKEN` / `TERRAFORM_BASE_URL`（默认 `https://registry.terraform.io`）|
| `ATLAS_RELEASE_NOTES_PAGE_ID` | `CONFLUENCE_RELEASE_NOTES_PAGE_ID` |
| `ATLAS_CACHE_VALKEY_URL` / `_CLIENT` / `_MAX_ENTRIES` / `_TTL_SECONDS` | `CACHE_VALKEY_URL` / `CACHE_VALKEY_CLIENT` / `CACHE_MAX_ENTRIES` / `CACHE_TTL_SECONDS` |

**新增（旧仓库没有）**：
- `CONFLUENCE_GUIDANCE_ONBOARDING_PAGE_ID`（onboarding guidance 页 id）
- `CONFLUENCE_AVAILABILITY_PAGE_<ID>`（每个 LZ 一个，如 `CONFLUENCE_AVAILABILITY_PAGE_AWSF` / `_AZURE`）
- `CONFLUENCE_SPACE_KEYS`（逗号分隔，reference discovery 用）
- `CONFLUENCE_SECURITY_BASE_URL` / `_TOKEN` / `_EMAIL` / `_SPACE_KEY`（安全策略空间，**可能是独立 Confluence 实例**；未配则 fallback 到主 `CONFLUENCE_*`）

**读法变化**：page id 现在**按调用惰性读取**（如 `releaseNotesPageId()`），不再是模块加载常量。

### 1.2 被删除的实体 / 契约（旧代码里凡引用到都要退役）

| 退役 | 替代 |
|---|---|
| `Topic` / `TopicSchema` / `topicTypes` / topicRepository / topic 路由 | `ResourceContextRecord` / `ResourceRecordResponse`，catalog 直接读 discovery 派生的 resource |
| `Anchor` / `AnchorSchema` / `anchorStrategies` / resolveAnchor / anchorRepository | 扁平 `heading?` + `selector?`（运行时 slug 匹配 heading）|
| `SourceTopicMapping` / mapping 仓库 | `SECTION_RULES` + discovered headings 在 derive 时直接绑定 section→source |
| `ContextBundle*` / `ContextRequest` / disclosure_level / expansion_paths | `ResourceContextResponse`，Section 带 `status` + `warnings[]` |
| `registryManifest.ts`（Sources/Topics/Anchors/Mappings 跨文件校验）| 无替代；resource 由 discovery 派生，不再有 manifest gate |
| `pilotRegistry.ts` / `loadRegistryFromManifests` / `data/*.yaml` seed | discover→derive 管线（唯一硬编码根 = `LANDING_ZONES`）|
| `pilotSourceContent.ts`（内存种子）| live provider + `devMocks/` MSW fixtures |
| `loadReleaseNotes` / `loadAnnouncements`（YAML loader）| `parseReleaseNotes` / `parseAnnouncements`（解析 Confluence 页面渲染文本）|

### 1.3 解析上下文 & honest-gap

- `offlineResolutionContext()` → `defaultResolutionContext()`：`fetch` 现在**晚绑定**（每次读 `globalThis.fetch`，为了 dev MSW / prod real 同一路径）。
- `ResolutionContext` 新增 `pageCache?: Map<string, Promise<unknown>>`（请求级 fetch+parse 记忆化，单飞）和 `scope?: { landingZoneId?; appId? }`（可见性 seam）。
- **honest-gap 原则贯穿所有 provider**：未配置 channel / 抓取失败 / 页面不合规 → 返回空/`null`/`source_unavailable` warning，**绝不编造 fallback 数据、绝不拿别的 LZ 的数据顶替**。
